## Error Tracking and Monitoring

The scraper server includes built-in error tracking with Sentry integration for production monitoring.

## Sentry Integration

### Configuration

From `app.py:61-74`, Sentry is automatically configured if the DSN is set:

```python
if os.getenv('SENTRY_DSN'):
    sentry_sdk.init(
        dsn=os.getenv('SENTRY_DSN'),
        integrations=[
            FlaskIntegration(),
            CeleryIntegration()
        ],
        traces_sample_rate=0.1,  # 10% performance monitoring
        environment=os.getenv('FLASK_ENV', 'development'),
        release=os.getenv('APP_VERSION', '1.0.0')
    )
```

### Sentry Features

**Automatic Error Capture:**
- All unhandled exceptions in Flask routes
- All Celery task failures
- Background worker crashes
- Database connection errors

**Performance Monitoring:**
- 10% of transactions tracked (configurable via `traces_sample_rate`)
- API endpoint response times
- Database query performance
- Task execution duration

<Info>
  The `traces_sample_rate=0.1` means 10% of requests are sampled for performance monitoring to reduce overhead and costs.
</Info>

### Task Errors

<Accordion title="Task Retry Logic">

From `tasks.py:88-95`, all tasks inherit retry behavior:

```python
class BaseTask(Task):
    autoretry_for = (Exception,)
    retry_kwargs = {'max_retries': 3}
    retry_backoff = True
    retry_backoff_max = 600  # 10 minutes
    retry_jitter = True
```

**Retry Schedule:**
- Attempt 1: Immediate
- Attempt 2: ~2-4 seconds (with jitter)
- Attempt 3: ~8-16 seconds (with jitter)
- Attempt 4: ~32-64 seconds (with jitter)
- Max backoff: 10 minutes

**Monitor Retries:**

```bash
heroku logs --tail --dyno worker | grep "Retry\|retry"
```

</Accordion>

<Accordion title="SoftTimeLimitExceeded">

**Time Limits:**

From `celery_config.py:43-44`:

```python
task_time_limit=7200,       # 2 hours hard limit
task_soft_time_limit=6900,  # 1h 55m soft limit
```

**What Happens:**
- At 1h 55m: `SoftTimeLimitExceeded` raised
- Task can catch and cleanup
- At 2h: Task forcefully killed

**Error Handling:**

From `tasks.py:194-196`:

```python
except SoftTimeLimitExceeded:
    logger.error(f"[Job {job_id}] Batch {batch_number}: Task exceeded time limit")
    raise
```

**Solutions:**
- Reduce batch sizes
- Increase worker count (parallel processing)
- Split large jobs into smaller chunks

</Accordion>

**Automatic Status Update:**

From `tasks.py:200-213`, failed tasks update job status:

### Database Errors

<Accordion title="Connection Pool Exhaustion">

**Error Message:**
```
Error: Too many connections
Error: Connection pool exhausted
```

**Root Cause:**

Supabase free tier allows max 50 concurrent connections.

**Prevention:**

From `app.py:165-166`, singleton pattern prevents connection leaks:

```python
_supabase_client = None
_supabase_lock = threading.Lock()

def get_supabase_client() -> Client:
    global _supabase_client
    if _supabase_client is not None:
        return _supabase_client
```

**Solution:**

```bash
# Check current pool size
heroku config:get SUPABASE_POOL_SIZE

# Reduce if needed (free tier)
heroku config:set SUPABASE_POOL_SIZE=3

# Monitor connection count in Supabase dashboard
```

</Accordion>

<Accordion title="Duplicate Key Errors">

**Error Message:**
```
ERROR: duplicate key value violates unique constraint
```

**Handling:**

From `utils/batch_processor.py:131-141`, duplicates are gracefully handled:

```python
# Only add to global_usernames if it doesn't exist
if profile_id not in existing_ids:
    global_records.append({
        'id': profile_id,
        'username': username,
        'full_name': full_name,
        'used': False,
        'created_at': current_timestamp,
        'base_id': base_id
    })
else:
    skipped_existing += 1
```

**This is Normal:**
- Profiles may be scraped multiple times
- System tracks existing profiles to avoid duplicates
- No action needed - automatic handling

</Accordion>

## Error Analysis

### Error Patterns

<CodeGroup>

```sql Frequent Errors Query
-- In Sentry or logs, identify most common errors
SELECT error_message, COUNT(*) as occurrences
FROM error_logs
WHERE created_at > NOW() - INTERVAL '24 hours'
GROUP BY error_message
ORDER BY occurrences DESC
LIMIT 10;
```

```sql Failed Jobs Analysis
-- Analyze job failure patterns
SELECT 
    DATE_TRUNC('hour', created_at) as hour,
    status,
    COUNT(*) as job_count
FROM scrape_jobs
WHERE created_at > NOW() - INTERVAL '7 days'
GROUP BY hour, status
ORDER BY hour DESC;
```

```sql Error Rate by Platform
-- If using multi-platform scraping
SELECT 
    platform,
    COUNT(CASE WHEN status = 'failed' THEN 1 END) as failures,
    COUNT(*) as total,
    ROUND(100.0 * COUNT(CASE WHEN status = 'failed' THEN 1 END) / COUNT(*), 2) as failure_rate
FROM scrape_jobs
WHERE created_at > NOW() - INTERVAL '7 days'
GROUP BY platform;
```

</CodeGroup>

## Error Prevention

### Input Validation

```python
# Add validation to prevent errors
def validate_request_data(data):
    errors = []
    
    if not data.get('accounts'):
        errors.append('Missing accounts field')
    elif not isinstance(data['accounts'], list):
        errors.append('accounts must be a list')
    elif len(data['accounts']) > 50:
        errors.append('Max 50 accounts per request')
    
    if errors:
        raise ValueError('; '.join(errors))
```

### Rate Limiting

```python
# Already implemented in app.py:94-101
from flask_limiter import Limiter

limiter = Limiter(
    app=app,
    key_func=get_remote_address,
    default_limits=["200 per hour"],
    storage_uri='memory://'
)
```

### Graceful Degradation

```python
# Example: Fallback for Apify failures
try:
    followers = scrape_followers(accounts, max_count, platform)
except Exception as e:
    logger.error(f"Apify scraping failed: {e}")
    # Return empty result instead of crashing
    return {
        'job_id': job_id,
        'profiles': [],
        'warning': 'Scraping service temporarily unavailable'
    }
```
