## Debugging the Scraper Server

This guide provides debugging techniques and commands for diagnosing issues in the scraper server.

## Log Analysis

### Viewing Logs

<CodeGroup>

```bash Heroku
# Real-time log stream
heroku logs --tail

# Last 1000 lines
heroku logs --num 1000

# Worker logs only
heroku logs --tail --dyno worker

# Web dyno logs only
heroku logs --tail --dyno web

# Filter by date
heroku logs --tail --since="2025-01-01 00:00"
```

```bash Render
# Access via Render dashboard
# Services → Your Service → Logs

# Or use Render CLI
render logs -f
```

```bash Local Development
# Flask app logs
python app.py

# Celery worker logs
celery -A celery_config worker --loglevel=info
```

</CodeGroup>

### Log Patterns to Search For

**Error Detection:**

```bash
# Find all errors
heroku logs | grep ERROR

# Find warnings
heroku logs | grep WARNING

# Find Apify failures
heroku logs | grep "Apify.*error\|Failed to scrape"

# Find database errors
heroku logs | grep "Supabase\|SUPABASE\|database"

# Find memory issues
heroku logs | grep "R14\|R15\|Memory"
```

**Success Monitoring:**

```bash
# Track scraping progress
heroku logs --tail | grep "Scraped\|profiles"

# Monitor batch processing
heroku logs --tail | grep "Batch\|batch"

# Watch job completion
heroku logs --tail | grep "complete\|completed"
```

### Understanding Log Levels

From `app.py:51-56`, logging is configured with timestamps:

```python
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(name)s: %(message)s',
    datefmt='%Y-%m-%d %H:%M:%S'
)
```

**Log Levels:**
- `INFO`: Normal operations, progress updates
- `WARNING`: Non-critical issues (e.g., skipped profiles)
- `ERROR`: Critical failures requiring attention
- `DEBUG`: Detailed diagnostic info (not enabled in production)

**Task Retry Logic:**

From `tasks.py:88-95`:

```python
class BaseTask(Task):
    autoretry_for = (Exception,)
    retry_kwargs = {'max_retries': 3}
    retry_backoff = True
    retry_backoff_max = 600  # 10 minutes
    retry_jitter = True
```

Tasks automatically retry up to 3 times with exponential backoff.

</Accordion>

<Accordion title="Debug Redis Queue">

**Inspect Queue State:**

```bash

# Shows number of pending tasks in main queue

# Check all queues
> LLEN default
> LLEN scraping
> LLEN processing

# View pending tasks (first 10)
> LRANGE celery 0 9

# Clear queue (if needed)
> DEL celery
```

**Queue Configuration:**

From `celery_config.py:57-62`, three queues are configured:

```python
task_queues=(
```

**pdb Commands:**
- `n` - Next line
- `s` - Step into function
- `c` - Continue execution
- `p variable_name` - Print variable
- `l` - List current code
- `q` - Quit debugger

## Database Debugging

### Query Active Jobs

```sql
-- In Supabase SQL Editor
SELECT job_id, status, progress, profiles_scraped, created_at
FROM scrape_jobs
WHERE status IN ('queued', 'processing')
ORDER BY created_at DESC
LIMIT 10;
```

### Check for Stuck Jobs

```sql
-- Jobs processing for > 2 hours
SELECT job_id, status, created_at, updated_at,
       AGE(NOW(), created_at) as time_running
FROM scrape_jobs
WHERE status = 'processing'
  AND created_at < NOW() - INTERVAL '2 hours'
ORDER BY created_at;
```

### Monitor Global Usernames Pool

```sql
-- Check available unused profiles
SELECT base_id, 
       COUNT(*) as total,
       SUM(CASE WHEN used = false THEN 1 ELSE 0 END) as unused,
       SUM(CASE WHEN used = true THEN 1 ELSE 0 END) as used
FROM global_usernames
GROUP BY base_id;
```

## Common Debugging Scenarios

<AccordionGroup>
  <Accordion title="Profile Data Not Saving">
    
**Symptoms:** Scraping completes but no data in database.

**Debug Steps:**

```bash
# 1. Check if profiles reached aggregation
heroku logs | grep "Aggregating results"

# 2. Check batch insert logs
heroku logs | grep "BULK INSERT"

# 3. Query database directly
# In Supabase SQL Editor:
SELECT COUNT(*) FROM raw_scraped_profiles 
WHERE scraped_at > NOW() - INTERVAL '1 hour';

# 4. Check for errors during insert
heroku logs | grep "Failed to insert"
```

**Common Causes:**
- Missing `base_id` in request (`tasks.py:134-139`)
- Database RLS policies blocking insert
- Invalid profile data format

</Accordion>

## Debugging Checklist

<Steps>
  <Step title="Reproduce the Issue">
    Document exact steps that trigger the problem.
  </Step>
  
  <Step title="Collect Logs">
    ```bash
    heroku logs --num 1000 > debug.log
    ```
  </Step>
  
  <Step title="Check Configuration">
    ```bash
    heroku config > config.txt
    # Review all environment variables
    ```
  </Step>
  
  <Step title="Test Isolation">
    Test each component separately (API, database, Redis, Apify).
  </Step>
  
  <Step title="Review Code Path">
    Trace through relevant source files to understand flow.
  </Step>
  
  <Step title="Apply Fix">
    Make changes, test locally, then deploy.
  </Step>
  
  <Step title="Verify Solution">
    Monitor logs after deployment to confirm fix.
  </Step>
</Steps>
