The Click Creators Scraper Server uses sophisticated batch processing techniques to handle 500K+ profiles efficiently while staying within Supabase free tier limits. This guide covers the batch processing utilities and optimization strategies.

## Overview

Batch processing is critical for performance at scale:

- **Bulk Inserts**: Insert 1000 records in a single query instead of 1000 individual queries
- **Rate Limiting**: 100ms delays between batches prevent throttling
- **Connection Pooling**: Reuse database connections across operations
- **Graceful Fallback**: Automatically retry with individual inserts if batch fails

<Info>
  The system can ingest 100,000 profiles in ~2 minutes using batch processing vs. ~8 hours with individual inserts.
</Info>

## Batch Insert Profiles

The `batch_insert_profiles` function handles bulk profile ingestion with optimizations for the Supabase free tier.

### Function Signature

```python utils/batch_processor.py
def batch_insert_profiles(
    supabase: Client,
    profiles: List[Dict],
    base_id: str,
    batch_size: int = 1000,
    rate_limit_delay: float = 0.1
) -> Tuple[int, int, int]:
    """
    Insert profiles into Supabase using TRUE bulk inserts for maximum performance.
    
    Args:
        supabase: Supabase client instance
        profiles: List of profile dictionaries with id, username, full_name
        base_id: Multi-tenant identifier for data isolation (REQUIRED)
        batch_size: Number of profiles to insert per batch (default: 1000)
        rate_limit_delay: Delay in seconds between batches (default: 0.1 = 100ms)
        
    Returns:
        Tuple of (inserted_raw, added_to_global, skipped_existing)
    """
```

### Usage Example

```python
from utils.batch_processor import batch_insert_profiles
from supabase import create_client

supabase = create_client(url, key)

profiles = [
    {
        'id': '123456789',
        'username': 'john_doe',
        'full_name': 'John Doe'
    },
    {
        'id': '987654321',
        'username': 'jane_smith',
        'full_name': 'Jane Smith'
    },
    # ... up to 100,000 profiles
]

# Insert profiles with base_id for multi-tenant isolation
inserted_raw, added_to_global, skipped = batch_insert_profiles(
    supabase=supabase,
    profiles=profiles,
    base_id='appXYZ123ABC',
    batch_size=1000,
    rate_limit_delay=0.1
)

print(f"Raw profiles inserted: {inserted_raw}")
print(f"New global profiles: {added_to_global}")
print(f"Skipped (duplicates): {skipped}")
```

### How It Works

<Steps>
  <Step title="Profile Validation">
    Validate and prepare all profiles before processing:

    ```python
    valid_profiles = []
    for profile in profiles:
        # Validate required fields
        if 'id' not in profile or 'username' not in profile:
            logger.warning(f"Skipping profile with missing fields")
            continue
        
        valid_profiles.append({
            'id': str(profile['id']),
            'username': profile['username'],
            'full_name': profile.get('full_name', '')
        })
    ```
  </Step>

  <Step title="Check Existing Profiles">
    Use optimized bulk queries to check for duplicates:

    ```python
    # Check ALL existing profiles using PostgreSQL IN() in fewer queries
    all_profile_ids = [p['id'] for p in valid_profiles]
    existing_ids = set()
    
    # Use larger chunks (5000) with IN operator for efficiency
    chunk_size = 5000
    for i in range(0, len(all_profile_ids), chunk_size):
        chunk_ids = all_profile_ids[i:i + chunk_size]
        
        existing = supabase.table('global_usernames')\
            .select('id')\
            .in_('id', chunk_ids)\
            .execute()
        
        if existing.data:
            existing_ids.update(r['id'] for r in existing.data)
    ```

    <Note>
      For 100K profiles, this reduces database queries from 100,000 to just 20 (5000 IDs per query).
    </Note>
  </Step>

  <Step title="Prepare Batch Records">
    Separate profiles into raw inserts and new global profiles:

    ```python
    raw_records = []
    global_records = []
    
    for profile in valid_profiles:
        # Always add to raw_scraped_profiles (historical record)
        raw_records.append({
            'id': profile['id'],
            'username': profile['username'],
            'full_name': profile['full_name'],
            'scraped_at': current_timestamp,
            'base_id': base_id
        })
        
        # Only add to global_usernames if it doesn't exist
        if profile['id'] not in existing_ids:
            global_records.append({
                'id': profile['id'],
                'username': profile['username'],
                'full_name': profile['full_name'],
                'used': False,
                'created_at': current_timestamp,
                'base_id': base_id
            })
        else:
            skipped_existing += 1
    ```
  </Step>

  <Step title="Bulk Insert Raw Profiles">
    Insert into `raw_scraped_profiles` in batches of 1000:

    ```python
    # From batch_processor.py:146-172
    for i in range(0, len(raw_records), batch_size):
        batch = raw_records[i:i + batch_size]
        
        try:
            # Single INSERT with multiple records
            supabase.table('raw_scraped_profiles').insert(batch).execute()
            inserted_raw += len(batch)
            
            # Rate limiting protection for Supabase Free Tier
            if i + batch_size < len(raw_records):
                time.sleep(rate_limit_delay)  # 100ms delay
                
        except Exception as e:
            logger.error(f"Failed to insert batch: {str(e)}")
            # Try individual inserts as fallback
            for record in batch:
                try:
                    supabase.table('raw_scraped_profiles').insert(record).execute()
                    inserted_raw += 1
                    time.sleep(0.01)  # 10ms delay
                except Exception as e2:
                    logger.warning(f"Failed individual insert: {str(e2)}")
    ```
  </Step>

  <Step title="Bulk Insert Global Profiles">
    Insert new profiles into `global_usernames`:

    ```python
    # From batch_processor.py:174-201
    if global_records:
        for i in range(0, len(global_records), batch_size):
            batch = global_records[i:i + batch_size]
            
            try:
                supabase.table('global_usernames').insert(batch).execute()
                added_to_global += len(batch)
                
                if i + batch_size < len(global_records):
                    time.sleep(rate_limit_delay)
                    
            except Exception as e:
                # Fallback to individual inserts
                for record in batch:
                    try:
                        supabase.table('global_usernames').insert(record).execute()
                        added_to_global += 1
                        time.sleep(0.01)
                    except Exception as e2:
                        skipped_existing += 1
    ```
  </Step>
</Steps>

## Performance Characteristics

### Batch Size Optimization

The default batch size of 1000 is optimized for Supabase free tier:

| Batch Size | Payload Size | Free Tier Safe? | Performance |
|------------|--------------|-----------------|-------------|
| 100 | ~20 KB | Yes | Slow (too many requests) |
| 500 | ~100 KB | Yes | Good |
| **1000** | **~200 KB** | **Yes** | **Optimal** |
| 2000 | ~400 KB | Yes | Marginal improvement |
| 5000 | ~1 MB | Yes | Risk of timeouts |
| 10000 | ~2 MB | Risky | May hit payload limits |

<Note>
  Supabase free tier has an 8 MB payload limit. A batch of 1000 profiles (~200 KB) provides safety margin while maximizing throughput.
</Note>

### Rate Limiting Strategy

```python
# 100ms delay between batches
rate_limit_delay = 0.1  # seconds

# For 100,000 profiles:
# - 100 batches of 1000
# - 99 delays of 100ms = 9.9 seconds total delay
# - Actual insert time: ~30-60 seconds
# - Total time: ~40-70 seconds
```

<Info>
  The 100ms delay prevents rate limiting while adding minimal overhead (< 10 seconds for 100K profiles).
</Info>

### Implementation Notes

<Warning>
  Due to Supabase/PostgREST limitations, bulk updates still execute as individual UPDATE queries with batched error handling. For true bulk updates at PostgreSQL level, use stored procedures or RPC functions.
</Warning>

## Integration with Celery Tasks

Batch processing is integrated into Celery tasks for asynchronous execution:

### Profile Ingestion Task

```python tasks.py
@celery.task(base=BaseTask, bind=True, name='tasks.ingest_profiles_batch')
def ingest_profiles_batch(
    self,
    batch_id: str,
    profiles: List[Dict],
    batch_number: int = 1,
    base_id: str = None
) -> Dict[str, Any]:
    """
    Ingest a batch of profiles into Supabase (max 1000).
    """
    try:
        logger.info(f"Ingesting {len(profiles)} profiles (base_id={base_id})")
        
        # Set RLS context for multi-tenant isolation
        if base_id:
            from utils.rls_context import set_rls_context
            set_rls_context(base_id)
        
        supabase = get_supabase_client()
        
        # Use batch processor for insertion
        inserted_raw, added_to_global, skipped = batch_insert_profiles(
            supabase,
            profiles,
            base_id=base_id,
            batch_size=500
        )
        
        return {
            'batch_id': batch_id,
            'batch_number': batch_number,
            'inserted_raw': inserted_raw,
            'added_to_global': added_to_global,
            'skipped': skipped
        }
        
    except Exception as e:
        logger.error(f"Batch ingestion error: {str(e)}")
        raise
```

### Async Ingestion Endpoint

```python api_async.py
@app.route('/api/ingest', methods=['POST'])
@limiter.limit("30 per hour")
def ingest_profiles_async():
    """
    Queue profile ingestion job.
    """
    data = request.get_json()
    profiles = data['profiles']
    base_id = get_base_id_from_request()
    
    # Split profiles into batches of 1000
    batch_size = 1000
    profile_batches = [
        profiles[i:i + batch_size] 
        for i in range(0, len(profiles), batch_size)
    ]
    batch_id = str(uuid.uuid4())
    
    # Queue batch tasks
    for i, batch in enumerate(profile_batches, 1):
        ingest_profiles_batch.delay(
            batch_id=batch_id,
            profiles=batch,
            batch_number=i,
            base_id=base_id
        )
    
    return jsonify({
        'success': True,
        'batch_id': batch_id,
        'batch_count': len(profile_batches),
        'total_profiles': len(profiles)
    }), 202
```

## Free Tier Considerations

The batch processing system is designed to work within Supabase free tier limits:

### Connection Limits

- **Free Tier Limit**: 50 concurrent connections
- **Solution**: Thread-safe singleton Supabase client with connection pooling (tasks.py:30-85)
- **Configuration**: `SUPABASE_POOL_SIZE=5` (5 connections per worker)

```python tasks.py
_supabase_client = None
_supabase_lock = threading.Lock()

def get_supabase_client() -> Client:
    """
    Initialize and return Supabase client.
    Thread-safe singleton pattern to reuse client across Celery tasks.
    """
    global _supabase_client
    
    if _supabase_client is not None:
        return _supabase_client
    
    with _supabase_lock:
        if _supabase_client is not None:
            return _supabase_client
        
        _supabase_client = create_client(supabase_url, supabase_key)
        return _supabase_client
```

### Payload Size Limits

- **Free Tier Limit**: 8 MB per request
- **Batch Size**: 1000 records (~200 KB)
- **Safety Margin**: 40x below limit

### Rate Limiting

- **Strategy**: 100ms delay between batches
- **Impact**: < 10 seconds added for 100K profiles
- **Benefit**: Prevents throttling and ensures stability

## Error Handling

### Graceful Fallback

If a batch insert fails, the system automatically falls back to individual inserts:

```python
try:
    # Attempt bulk insert
    supabase.table('raw_scraped_profiles').insert(batch).execute()
    inserted_raw += len(batch)
except Exception as e:
    logger.error(f"Batch insert failed: {str(e)}")
    
    # Fallback: Try each record individually
    for record in batch:
        try:
            supabase.table('raw_scraped_profiles').insert(record).execute()
            inserted_raw += 1
            time.sleep(0.01)  # Small delay for individual inserts
        except Exception as e2:
            logger.warning(f"Individual insert failed: {str(e2)}")
            # Continue with next record
```

### Retry Logic

Celery tasks include automatic retry with exponential backoff:

```python tasks.py
class BaseTask(Task):
    """Base task with error handling and retry logic."""
    
    autoretry_for = (Exception,)
    retry_kwargs = {'max_retries': 3}
    retry_backoff = True
    retry_backoff_max = 600  # 10 minutes
    retry_jitter = True
```

## Best Practices

<AccordionGroup>
  <Accordion title="Choose appropriate batch sizes" icon="sliders">
    - Use 1000 for inserts (optimal payload size)
    - Use 500 for updates (more complex operations)
    - Use 5000 for read-only queries (checking existing records)
  </Accordion>
  
  <Accordion title="Always include rate limiting" icon="clock">
    - 100ms delay between batches prevents throttling
    - Adjust based on your tier and load
    - Monitor for 429 (Too Many Requests) errors
  </Accordion>
  
  <Accordion title="Implement graceful fallback" icon="life-ring">
    - Catch batch operation failures
    - Retry individual records to salvage partial data
    - Log failures for debugging
  </Accordion>
  
  <Accordion title="Use multi-tenant isolation" icon="lock">
    - Always pass `base_id` to batch operations
    - Set RLS context in Celery tasks
    - Isolate data between different clients
