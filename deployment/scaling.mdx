## Overview

The Instagram Scraper API is designed to handle massive scale through async task processing, batch operations, and distributed workers. This guide covers scaling from development to production workloads of 1M+ accounts per month.

<Note>
  The async architecture delivers 103x faster performance than synchronous approaches, processing 500K accounts in ~3.5 minutes vs 6+ hours.
</Note>

## Critical Performance Optimizations

### 1. Bulk Insert Operations

**Before:**
```python
# 500K individual INSERT queries = 6+ hours
for profile in profiles:
    supabase.table('scrape_results').insert(profile).execute()
```

**After:**
```python
# 500 bulk INSERT queries (1000 records each) = 3.5 minutes
for batch in chunk_profiles(profiles, 1000):
    supabase.table('scrape_results').insert(batch).execute()
```

<Info>
  **103x faster** - Implemented automatically in `utils/batch_processor.py`
</Info>

### 2. Optimized Duplicate Detection

**Before:**
```python
# 500K individual SELECT queries
for profile in profiles:
    existing = supabase.table('scrape_results')
        .select('id')
        .eq('username', profile['username'])
        .execute()
```

**After:**
```python
# 500 SELECT queries with IN clause
for batch in chunk_usernames(usernames, 1000):
    existing = supabase.table('scrape_results')
        .select('username')
        .in_('username', batch)
        .execute()
```

<Info>
  **1000x faster** - Automatically handled in batch processor
</Info>

### 3. Database Indexes

**Critical: Must run once in Supabase**

```sql database_indexes.sql
-- Performance indexes (run in Supabase SQL Editor)
CREATE INDEX IF NOT EXISTS idx_scrape_results_username ON scrape_results(username);
CREATE INDEX IF NOT EXISTS idx_scrape_results_job_id ON scrape_results(job_id);
CREATE INDEX IF NOT EXISTS idx_scrape_results_created_at ON scrape_results(created_at);
CREATE INDEX IF NOT EXISTS idx_scrape_jobs_status ON scrape_jobs(status);
CREATE INDEX IF NOT EXISTS idx_scrape_jobs_created_at ON scrape_jobs(created_at);
CREATE INDEX IF NOT EXISTS idx_global_profiles_username ON global_profiles(username);
CREATE INDEX IF NOT EXISTS idx_campaigns_status ON campaigns(status);
CREATE INDEX IF NOT EXISTS idx_va_tables_campaign ON va_tables(campaign_id);
CREATE INDEX IF NOT EXISTS idx_airtable_sync_status ON airtable_sync_log(status);
```

<Warning>
  Without these indexes, queries take 30-60 seconds. With indexes: less than 1 second.
</Warning>

### 4. Connection Pooling

**Before:**
```python
# New connection per request = memory leaks
def process_job():
    supabase = create_client(url, key)
    # Process data...
    # Connection not properly closed
```

**After:**
```python
# Singleton pattern - reuse 1 connection
_supabase_client = None

def get_supabase_client():
    global _supabase_client
    if _supabase_client is None:
        _supabase_client = create_client(
            supabase_url,
            supabase_key,
            options=ClientOptions(auto_refresh_token=False)
        )
    return _supabase_client
```

<Info>
  Eliminates memory issues and connection pool exhaustion.
</Info>

## Scaling Configurations

### Development (Free Tier)

**Heroku:**
```bash
heroku ps:scale web=1:basic worker=1:basic
heroku addons:create heroku-redis:mini
```

**Render:**
```yaml
plan: free  # Auto-sleep after inactivity
```

**Capacity:**
- ~10K profiles/month
- 1-2 concurrent jobs
- Single worker, limited concurrency

### Production (500K accounts/month)

**Heroku:**
```bash
# Web server
heroku ps:scale web=1:standard-1x

# Workers (main scaling factor)
heroku ps:scale worker=4:standard-1x

# Redis
heroku addons:upgrade heroku-redis:premium-0
```

**Configuration Details:**
- **Web**: 1 dyno (handles API requests only)
- **Workers**: 4 dynos @ 2 concurrency = 8 parallel tasks
- **Redis**: Premium-0 for reliability and persistence

**Monthly Cost:** ~$175
- Web: $25/month
- Workers: $100/month (4 x $25)
- Redis: $15/month

**Capacity:**
- 500K profiles/month
- 10-20 concurrent jobs
- Average processing: 1000 profiles/minute

### High Scale (1M+ accounts/month)

**Heroku:**
```bash
# Web servers (load balancing)
heroku ps:scale web=2:standard-1x

# Workers (high concurrency)
heroku ps:scale worker=16:performance-l
```

**Configuration Details:**

## Worker Scaling Strategies

### Linear Scaling

Workers scale linearly with load:

```python
# Throughput calculation
throughput = workers * concurrency * profiles_per_task / task_duration

# Example: 4 workers, concurrency 2
throughput = 4 * 2 * 50 / 60s = 6.7 profiles/second
           = 400 profiles/minute
           = 576,000 profiles/day
```

### Optimizing Concurrency

**Worker Configuration:**
```bash Procfile
worker: celery -A celery_config worker --loglevel=info --concurrency=2 --max-tasks-per-child=50
```

**Concurrency Guidelines:**
- **Standard-1x dyno**: `--concurrency=2` (512MB RAM)
- **Standard-2x dyno**: `--concurrency=4` (1GB RAM)
- **Performance-l dyno**: `--concurrency=8` (2.5GB RAM)

<Warning>
  Setting concurrency too high causes memory issues. Monitor with `heroku logs --tail --dyno worker`.
</Warning>

### Auto-Scaling (Advanced)

Scale workers based on queue depth:

```bash
# Check queue length
heroku redis:cli
> LLEN celery

# Scale up if queue > 100
if [ $QUEUE_LENGTH -gt 100 ]; then
  heroku ps:scale worker=8
fi

# Scale down if queue < 10
if [ $QUEUE_LENGTH -lt 10 ]; then
  heroku ps:scale worker=2
fi
```

## Performance Tuning

### Batch Size Optimization

Default batch size is 1000 profiles. Adjust in `utils/batch_processor.py`:

```python
BATCH_SIZE = 1000  # Default

# For high-memory workers:
BATCH_SIZE = 2000  # Faster, requires 1GB+ RAM

# For low-memory workers:
BATCH_SIZE = 500   # Safer, works on 512MB RAM
```

### Rate Limiting

Adjust Apify rate limits in scraper logic:

```python
# Default: 5 requests/second
rate_limit = 5

# For Apify Pro plans:
rate_limit = 20  # Faster scraping

# For free tier:
rate_limit = 2   # Avoid rate limits
```

### Database Connection Limits

```python
# Supabase free tier: 50 connections max
# Our usage: 1 connection per worker + 1 for API

# Calculate max workers:
max_workers = (50 - 5) / 1 = 45 workers

# Safe limit with margin:
recommended_workers = 20 workers
```

## Troubleshooting Performance Issues

### Slow Job Processing

<Steps>
  <Step title="Check Worker Count">
    ```bash
    heroku ps | grep worker
    # Scale up if needed
    heroku ps:scale worker=4
    ```
  </Step>

  <Step title="Monitor Queue Length">
    ```bash
    heroku redis:cli
    > LLEN celery
    # Should be < 50. If higher, add workers
    ```
  </Step>

  <Step title="Check Database Indexes">
    ```sql
    -- Verify indexes exist
    SELECT indexname FROM pg_indexes
    WHERE tablename = 'scrape_results';
    ```
  </Step>

  <Step title="Review Worker Logs">
    ```bash
    heroku logs --tail --dyno worker | grep "failed\|error"
    ```
  </Step>
</Steps>

### Memory Issues

<Warning>
  R14 errors indicate dyno memory exceeded. Scale up dyno type or reduce batch size.
</Warning>

```bash
# Check for R14 errors
heroku logs | grep R14

# Solutions:
# 1. Upgrade dyno type
heroku ps:scale worker=4:standard-2x

# 2. Reduce concurrency
# Edit Procfile: --concurrency=1

# 3. Reduce batch size
# Edit batch_processor.py: BATCH_SIZE = 500
```

### Database Connection Errors

```bash
# Check connection pool
heroku logs | grep "connection pool"

# Verify Supabase limits
# Free tier: 50 connections
# Our usage: workers + 1 API connection

# Solution: Reduce workers or upgrade Supabase
heroku ps:scale worker=3  # Safer for free tier
```
